# 知识库钩子 P0 收紧与检索提准设计

## 1. 背景

当前知识库钩子的生命周期安全、命令文档单一来源和 CI 门禁已经比较完整，但实际使用暴露出三类问题：

- `parseOptions` 对未知参数和重复参数默认忽略或覆盖，写命令的拼写错误可能改变业务语义而不报错。
- `check-stale`、`refresh-project` 和 `promote` 没有共用一套知识库真实路径校验，直接 CLI 可以传入知识库外文件。
- `before-task` 只按 `score >= 8` 判断必读。中文二元词和同义词被当成相互独立的关键词重复计分，正文零散命中容易产生大量必读误报。

真实知识库当前只有 8 个活跃 Markdown；“查询章节图谱聚合接口数据源不一致”返回 7 个必读，“优化知识库钩子检索性能”也把业务项目知识判成必读。因此本阶段优先解决输入安全和必读精度，不先增加索引或 MCP。

## 2. 目标

- 所有命令只接受自身声明的全局选项、命令选项和位置参数；未知、重复、缺值参数在产生副作用前失败。
- `check-stale`、`refresh-project` 只读取或修改当前知识库 `knowledge/` 下的普通 Markdown 文件，并要求 `status: confirmed`。
- `promote` 只读取当前知识库 `inbox/` 下的普通 Markdown 文件，拒绝真实路径越界和文件符号链接。
- 同一同义词组只贡献一次字段得分和一次覆盖率，大小写变体不重复计分。
- JSON 同时返回原始查询词、实际扩展词、命中词、命中原因和必读原因。
- 必读必须是已确认知识且达到高覆盖率；正文弱命中默认进入“可能相关”；单次最多返回 5 个必读。
- 用真实任务风格的回归用例锁定误报边界。

## 3. 非目标

- 不增加持久化索引、文件监听、向量检索或 MCP Server。
- 不迁移 `ak.ps1` 到 Node，不拆分现有主 CLI 和大测试文件。
- 不修改知识库 Markdown 数据格式，不批量改写 `team-agent-knowledge`。
- 不改变 `record-fix`、`resolve-fix` 的纠偏生命周期。
- 不自动提交或推送。

## 4. 方案比较与决策

### 4.1 在现有 CLI 内做小范围闭环收紧（采用）

保留当前命令路由和零依赖实现，增加严格调用校验、共享知识文件解析 helper 和查询词组模型。优点是改动可由现有 Node 测试完整覆盖，失败面清楚；缺点是主文件暂时继续偏大。

### 4.2 先迁移为配置驱动路由（暂不采用）

把命令参数 schema、路由和 PowerShell 全部迁到 Node 能进一步消除重复，但会把安全修复、行为变化和架构迁移混在同一批次，回归范围过大。

### 4.3 先增加索引或 MCP（不采用）

索引只能减少扫描时间，不能解决必读误报；MCP 会把当前接口直接固化成服务契约。两者都应建立在本阶段行为稳定之后。

## 5. 命令调用校验

`parseGlobalOptions` 继续负责提取全局选项，但主分发在任何命令逻辑前调用统一校验：

- 所有全局选项最多出现一次。
- `--json` 只允许 `before-task`、`search`、`check-stale`、`doctor`。
- `--repository-root` 只允许 `sync-adapters`、`doctor`、`sync-command-docs`。
- `--knowledge-root` 允许知识库业务命令和 `doctor`；`sync-adapters` 为兼容会统一注入该参数的既有 `ak.ps1` 入口而接受但忽略它，模板来源仍只取 `--repository-root`；命令文档同步拒绝该参数。

命令级参数解析使用明确 schema，而不是解析 `command-contract.json` 中仅供展示的 `args` 文本。schema 支持：单值选项、多词值选项、布尔标记、必填项；统一拒绝未知参数、重复参数、缺值和多余位置参数。`search` / `before-task` 的独立 `--xxx` 形式参数也视为未知参数；如查询正文确实包含该文本，应把完整描述作为一个带引号的参数传入。

## 6. 知识文件路径闭环

新增共享解析 helper，按以下顺序校验：

1. 将相对路径闭环解析到知识库根，绝对路径也必须词法位于根内。
2. 文件扩展名必须为 `.md`。
3. `lstat` 必须是普通文件且不能是文件符号链接。
4. 知识库根和文件 `realpath` 之间仍保持包含关系，拒绝目录链接逃逸。
5. 调用方声明允许的顶层目录：`knowledge/` 或 `inbox/`。
6. `check-stale`、`refresh-project` 读取后确认 `status: confirmed`。

helper 返回原请求文件路径、规范相对路径和读取结果需要的安全真实路径；调用方不再自行拼接路径。

## 7. 检索词组和评分

查询模型包含：

- `queryTerms`：`extractKeywords` 得到的原始查询词。
- `groups`：原始词、中文二元词及其同义词构成的等价词组。
- `expandedTerms`：所有实际参与匹配的词，按 NFKC、半角和小写去重，但保留首个展示形式。

同一词组在文件名、标题、frontmatter、正文中各最多计分一次。覆盖率改为“命中的词组数 / 总词组数”，不会因为 `Graph/graph` 或 `queue/Queue` 重复膨胀。

## 8. 必读判定

文件成为必读必须满足：

- 位于 `knowledge/`；
- 覆盖率至少为 `0.5`；
- 至少命中一个词组；
- 并且标题、文件名或 frontmatter 有强命中，或者正文覆盖率至少为 `0.6` 且至少命中两个词组。

按现有相关性排序后最多保留 5 个必读，其余结果进入“可能相关”。JSON 和文本输出都读取同一 `result.mustRead`，不再分别重算阈值。

## 9. 输出兼容

保留现有 `keywords`、`coverage`、`matched`、`total`、`hits`、`mustRead` 字段；其中 `keywords` 改为真实扩展词列表。新增：

- `queryTerms`
- `expandedTerms`
- `matchedTerms`
- `reasonCodes`
- `mustReadReason`

已有消费者继续读取旧字段不会报错；需要解释相关性的新消费者使用新增字段。

## 10. 测试与验收

- 对每项新行为先写失败测试并确认失败原因正确。
- 参数测试必须断言失败前没有创建 `inbox/`、`knowledge/`、`work/` 或 `archive/` 工件。
- 路径测试覆盖绝对越界、目录链接逃逸、文件链接、非 confirmed 文件和合法知识文件。
- 检索测试覆盖同义词大小写去重、词组覆盖率、最多 5 个必读、正文弱命中降级和 JSON 解释字段。
- 全量 `npm.cmd test`、真实 `doctor`、两条示例任务回归和 UTF-8 BOM 检查全部通过。
