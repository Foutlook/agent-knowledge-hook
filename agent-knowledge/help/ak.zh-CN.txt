# ak 短命令帮助

`ak` 是 `agent-knowledge` 的 PowerShell 短命令包装器，用来把很长的底层命令收起来。

知识库位置解析顺序：

1. 优先使用环境变量 `AGENT_KNOWLEDGE_ROOT`
2. 如果没有环境变量，自动寻找同级目录 `team-agent-knowledge`
3. 如果仍找不到，使用工具内置示例知识库

常用查看方式：

```powershell
ak help
ak help check
ak help refresh
ak check --help
```

命令总览：

以下结构化命令区块由命令契约自动生成，请勿手工修改。

以下只读命令支持 `--json`：

```powershell
ak task "分析实体图谱 ownerId 为空" --json
ak search "聚合接口 数据源一致" --json
ak check poseidon --json
ak doctor --json
```

底层 CLI 会严格校验参数；未知参数、重复参数、缺失参数值和当前命令不支持的全局参数都会在执行前报错。

## task

用法：

```powershell
ak task "分析实体图谱 ownerId 为空"
```

作用：

任务开始前根据任务描述检索团队知识库，输出“必须阅读项”和“可能相关项”。同义词按查询组计算，组内别名只计一次覆盖率；只有高覆盖、高置信的已确认知识进入必须阅读，最多 5 项。

什么时候用：

- 开始分析需求、BUG、技术方案或代码评审前
- 不确定某个问题是否有历史坑或团队规则时
- Codex / Claude / OpenCode 进入正式分析前

是否写文件：

否。只读取知识库，不修改任何文件。

底层动作：

```text
agent-knowledge before-task <任务描述>
```

## search

用法：

```powershell
ak search "聚合接口 数据源一致"
```

作用：

按关键词搜索知识库，返回命中的规则、项目说明、服务关系、历史坑或纠错记录。结果按查询组覆盖率优先排序，并附带实际命中词和命中字段原因。

什么时候用：

- 临时查某个规则是否已经沉淀
- 想找某个服务、接口、模块、字段或业务口径的说明
- 不在完整任务流程中，只想快速查知识

是否写文件：

否。只读取知识库，不修改任何文件。

底层动作：

```text
agent-knowledge search <关键词>
```

## projects

用法：

```powershell
ak projects
```

作用：

读取 `knowledge/service-map/workspace-projects.md`，列出当前知识库已登记的项目名和本地路径。

什么时候用：

- 不记得项目名该写 `poseidon`、`reasearch-hub` 还是别的名字
- 新增仓库前，确认它是否已经登记
- 让 Codex 根据项目索引选择正确入口

是否写文件：

否。只读取项目索引。

## check

用法：

```powershell
ak check poseidon
```

作用：

检查某个项目知识文件是否落后于该项目当前 Git HEAD。

它会自动解析：

- 项目路径：来自项目知识文件 `project_root`，或项目索引 `workspace-projects.md`
- 知识文件：默认匹配 `knowledge/domain/project-<项目名>.md`
- 安全边界：只接受当前知识库 `knowledge/` 下、`status: confirmed` 的真实 Markdown 普通文件，不跟随越出知识库真实根的 symlink / junction

什么时候用：

- 开始处理某个项目相关需求前
- 项目刚合并 master / main 后
- 怀疑知识库里的项目说明已经落后于代码
- 新增模块、接口、Facade、Mapper、前端入口、依赖关系后

是否写文件：

否。只对比 `last_scanned_commit` 和当前项目 HEAD。

输出怎么看：

- “已是最新”：知识文件记录的 commit 与项目当前 HEAD 一致
- “可能过期”：项目代码已经变化，需要人工或 Codex 核对正文
- “缺少 last_scanned_commit”：该项目知识还没有建立刷新基线

底层动作：

```text
agent-knowledge check-stale --project-root <项目路径> --knowledge-file <知识文件>
```

## refresh

用法：

```powershell
ak refresh poseidon "同步本次需求合并后的模块变化"
```

作用：

在项目知识正文已经由人工或 Codex 核对后，刷新该知识文件的元数据。

它会更新：

- `updated`
- `project_root`
- `last_scanned_commit`
- 正文末尾的“刷新记录”

什么时候用：

- 已经根据最新代码更新完项目知识正文
- 确认项目说明已经覆盖本次合并后的变化
- 新增仓库完成初始项目知识文件后

是否写文件：

是。会修改指定项目知识文件。

重要边界：

`ak refresh` 不会自动重写项目说明正文。它只表示“这份知识已经核对到当前 commit”。如果正文没有更新，不能只靠 refresh 假装知识已经最新。

目标必须是当前知识库 `knowledge/` 下、`status: confirmed` 的真实 Markdown 普通文件；写锁取得后还会重新校验真实路径。

底层动作：

```text
agent-knowledge refresh-project --project-root <项目路径> --knowledge-file <知识文件> --summary <说明>
```

## bug

用法：

```powershell
ak bug "学习报告统计口径错误" --target knowledge/rules/learning-report.md
```

作用：

创建一条 BUG 修复纠错记录模板，写入 `inbox/fixes/`。如果被纠正对象是未确认草稿，应直接修改原草稿，不创建本记录。

什么时候用：

- BUG 分析过程中被纠正
- 最终修复原因和一开始判断不一致
- 发现某个历史坑会反复导致同类 BUG
- 修复后需要把失败点、调用链、原因、修复和验证方式沉淀下来

是否写文件：

是。写入 `inbox/fixes/`。

底层动作：

```text
agent-knowledge record-fix --type bug --title <标题> [--target <知识文件>]
```

带 `--target` 时，先修改并审核目标，再用 `ak resolve <生成的inbox文件>` 关闭，绝不能 `promote`；不带 `target` 的独立 fix 才能在确认应成为独立知识后 `promote`。

什么时候用：

- 需求范围被澄清
- 字段含义、统计口径、流程边界被纠正
- PRD 描述和真实业务规则不一致

是。写入 `inbox/prd-corrections/`。

底层动作：

```text
agent-knowledge record-fix --type prd --title <标题> [--target <知识文件>]
```

带 `--target` 时，先修改并审核目标，再用 `ak resolve <生成的inbox文件>` 关闭，绝不能 `promote`；不带 `target` 的独立 fix 才能在确认应成为独立知识后 `promote`。

## tech

用法：

```powershell
ak tech "图谱接口不能混用 ACL PT 集合和教研 PT 映射" --target knowledge/rules/aggregation-data-source-consistency.md
```

作用：

创建一条技术方案纠偏记录模板，写入 `inbox/tech-solution-corrections/`。如果被纠正对象是未确认草稿，应直接修改原草稿，不创建本记录。

什么时候用：

- 技术方案的数据源、接口边界、性能策略或依赖方向被纠正
- 发现原方案会造成 N+1、数据源不一致、错误 fallback 或服务边界混乱
- 需要把评审结论沉淀成后续可复用经验

是否写文件：

是。写入 `inbox/tech-solution-corrections/`。

纠错生命周期边界：

- `inbox/` 中尚未确认的 `draft` / `pending` 文件被纠正时，直接修改原文件。
- 已确认正式知识被纠正时，创建带 `--target` 的 targeted fix；独立业务结论被纠正时，创建不带 `target` 的独立 fix。
- targeted fix 创建后，先由人工或 Codex 修改并审核目标，再执行 `ak resolve`；不能执行 `ak promote`。
- 独立 fix 只有在确认应成为独立长期知识后，才执行 `ak promote`。
- 已知被纠正文件时可传 `--target <知识文件>`；如果目标是未确认草稿，命令会拒绝并提示直接修改。
- 不得让同一结论同时以原草稿和 fix 两种待确认形态重复存在。

底层动作：

```text
agent-knowledge record-fix --type tech --title <标题> [--target <知识文件>]
```

## resolve

用法：

```powershell
ak resolve inbox/tech-solution-corrections/<纠偏文件>.md
ak resolve inbox/tech-solution-corrections/<旧纠偏文件>.md --confirm-legacy
```

作用：

关闭已经由目标正式知识吸收的 targeted fix。命令校验目标仍是 `confirmed` 且当前 SHA-256 与记录时的 `target_hash` 不同，然后把纠偏 source 转换成可恢复、可审计的归档工件。命令不会自动修改或语义合并目标正文。

什么时候用：

1. 已阅读 fix 的失败点、证据链和纠偏结论。
2. 已由人工或获授权的 Codex 工作流修改 `target` 指向的正式知识。
3. 已完整审核目标正文，确认纠偏被正确、完整地吸收。

哈希变化只能证明目标文件字节发生过变化，不能证明语义正确；不能把 `resolve` 的哈希校验替代人工审核。

是否写文件：

是。成功后旧 inbox source 消失，并生成：

- `archive/source-survivors/<分类>/<文件>`：source survivor，与处理中 source 使用同一 inode，承接旧文件句柄的晚到写入。
- `archive/resolved-sources/<分类>/<文件>`：source snapshot，独立 inode 的只读字节快照，固定本次审计采用的原文。
- `archive/resolved/<分类>/<文件>`：resolved 审计记录，保存 source hash、目标基线/关闭哈希和两份 source 工件路径。

legacy 边界：

旧版 targeted fix 缺少 `target_hash` 时默认拒绝。只有人工已经确认目标吸收纠偏后，才能显式传 `--confirm-legacy`；该参数不是绕过内容审核的快捷方式。

失败与恢复：

- 命令会保留 `work/` claim、survivor、snapshot 或 resolved 等恢复现场；使用同一个 source 路径重试。
- 工件冲突、内容不一致或 source 路径复用时，保留全部版本并人工审核，不要手工删除、移动或晋升这些工件。
- `archive/` 与 `work/` 不参与知识检索、必须阅读清单或待确认清单，也不是第二份正式知识。

底层动作：

```text
agent-knowledge resolve-fix --file <inbox纠偏文件> [--confirm-legacy]
```

## rule

用法：

```powershell
ak rule "聚合接口实体集合和映射来源必须一致"
ak rule "禁止在循环中逐条远程查询" --confirmed
```

作用：

新增团队规则。

默认行为：

不带 `--confirmed` 时写入 `inbox/rules/`，表示待确认规则草稿。

确认行为：

带 `--confirmed` 时写入 `knowledge/rules/`，表示已经确认的长期规则。

什么时候用：

- 发现团队不成文规则
- 发现某类问题需要以后强约束
- 代码评审、BUG 修复或技术方案纠偏后形成可复用原则

是否写文件：

是。默认写入 `inbox/rules/`；带 `--confirmed` 写入 `knowledge/rules/`。

底层动作：

```text
agent-knowledge add-rule <规则标题> [--confirmed]
```

## promote

用法：

```powershell
ak promote inbox/rules/2026-07-09-aggregation-rule.md
```

作用：

将 `inbox/` 下已经人工确认的普通草稿或不带 `target` 的独立 fix 移动到 `knowledge/` 对应子目录，并把 frontmatter 的 `status` 改为 `confirmed`。

映射规则：

```text
inbox/<子目录>/<文件名>.md -> knowledge/<子目录>/<文件名>.md
```

什么时候用：

- `inbox/rules/` 中的规则草稿已经确认长期有效
- 不带 `target` 的独立 fix 已整理完毕，并确认应成为独立长期知识
- 定期清理待确认区，避免 inbox 长期堆积

是否写文件：

是。会写入 `knowledge/` 目标文件，并删除原 `inbox/` 文件。

安全边界：

- 只接受 `inbox/` 下的文件
- 源文件必须是真实 Markdown 普通文件，目标目录链不能包含 symlink / junction
- 如果目标 `knowledge/` 文件已经存在，会直接报错，不会覆盖已有已确认知识
- 带非空 `target` 的 targeted fix 会在任何写入前拒绝；必须先修改并审核目标，再执行 `ak resolve`

底层动作：

```text
agent-knowledge promote --file <inbox文件>
```

## pending

用法：

```powershell
ak pending
```

作用：

列出 `inbox/` 下仍待确认的知识条目，输出相对路径、`status`、`type` 和 `updated`。

什么时候用：

- 检查是否有长期未处理的规则草稿或纠错记录
- 准备晋升普通草稿/独立 fix，或关闭 targeted fix 前
- 定期维护团队知识库时

是否写文件：

否。只读取 inbox，不修改文件。

说明：

只有带 `status` frontmatter 的 Markdown 才会被列为待确认条目，`inbox/README.md` 这类说明文档不会出现在清单里。

底层动作：

```text
agent-knowledge list-pending
```

## adapters

用法：

```powershell
ak adapters
ak adapters --check
```

作用：

同步 OpenCode 命令适配器，或用 `--check` 只读检查目标是否缺失、是否与模板漂移。

模板边界：

- 唯一模板来源是工具仓库的 `agent-knowledge/templates/opencode/`。
- 安装目标是同一工具仓库的 `.opencode/command/`。
- 私有知识库的 `--knowledge-root` 不会改变模板或安装目标。
- `--check` 不写文件，发现漂移时退出 1；同步模式会更新目标文件。

底层动作：

```text
agent-knowledge sync-adapters [--check] --repository-root <工具仓库根目录>
```

## doctor

用法：

```powershell
ak doctor
ak doctor --json
```

作用：

只读扫描 `knowledge/` 与 `inbox/` 下的 Markdown，精确跳过 `inbox/README.md`。检查 UTF-8 BOM、frontmatter、目录对应的 `status`、全局重复标题、`target`、targeted fix 的 `target_hash` / `fix_id`、`project_root` / 逗号分隔的 `evidence_files`，以及 OpenCode 适配器漂移。还会只读诊断 `knowledge/`、`inbox/` 下的相邻 `*.md.lock` / `*.md.lock.reclaim`，以及固定 `work/locks/resolve/` 目录中名称为 64 位小写十六进制哈希的锁；其它 `.lock` 外观文件不读取。合法内容必须完整匹配 `PID:RFC4122-UUID`，末尾可以没有换行或只有一个 LF / CRLF。锁扫描不跟随 symlink / junction，也不会越出知识库真实根。只有工具仓库的 `.opencode/command/` 目录存在时才执行适配器检查，模板固定来自 `agent-knowledge/templates/opencode/`。`archive/` 与 `work/` 不进入知识正文扫描，`work/` 仅检查上述固定锁目录。

严重级别与退出码：

- `error`：BOM、frontmatter 缺失、状态非法、target 失效、`target_hash` 非法、`fix_id` 非法、适配器漂移。存在 error 时退出 1。
- `warning`：重复标题、缺少 project_root、证据文件无效或不存在、targeted fix 缺少 `target_hash`、owner PID 已退出的 `orphan_lock`、token 无法解析的 `invalid_lock`。只有 warning 时退出 0；缺 `target_hash` 表示关闭旧记录时需要显式 `--confirm-legacy`。
- 文本和 JSON 都会给出检查文件数及问题明细；`--json` 在退出 0 或 1 时 stdout 都只有一个合法 JSON 对象。
- 不会自动修复、晋升、删除或重写文件。

锁恢复边界：

- 普通主锁只有在调用成功持有相邻 reclaim guard、内容严格匹配 `PID:RFC4122-UUID` 且 owner PID 已退出时才会自动恢复；PID-only、BOM、额外内容或非法 UUID 不会被删除。
- orphan reclaim guard 不自动删除，以避免多个回收者之间的 ABA 误删；后续获取会安全超时并保留 guard。
- `LOCK_TIMEOUT` 不代表可以直接删锁。应先停止或确认没有相关任务，再用 `doctor` 核对路径和 PID，检查锁内容及相邻恢复工件，最后由团队协调人工处理；无法确认 token 所属任务时不要强制解锁。

参数边界：

只允许不带参数，或带唯一的 `--json`。包装器会显式传入工具仓库根目录，并继续按既有规则解析知识库根目录。

底层动作：

```text
agent-knowledge doctor [--json] --repository-root <工具仓库根目录>
```

## json

用法：

```powershell
ak task "分析实体图谱 ownerId 为空" --json
ak search "聚合接口 数据源一致" --json
ak check poseidon --json
ak doctor --json
```

作用：

让查询类命令输出 JSON，便于 Codex、OpenCode 或其它自动化脚本直接解析。`task` / `search` 会返回 `queryTerms`、`expandedTerms`，以及每项结果的 `mustReadReason`、`matchedTerms`、`reasonCodes` 等可解释字段；兼容字段 `keywords` 保留。

支持范围：

- `ak task`
- `ak search`
- `ak check`
- `ak doctor`

是否写文件：

否。只改变输出格式。

## raw

用法：

```powershell
ak raw search "RPC 本地依赖"
ak raw before-task "分析 graph-service 实体归属"
```

作用：

透传到底层 `agent-knowledge` CLI。适合临时使用还没有封装成短命令的能力。

什么时候用：

- 调试底层 CLI
- 使用还没有短命令别名的新能力
- 复现 README 中的原始命令

是否写文件：

取决于透传的底层命令。
