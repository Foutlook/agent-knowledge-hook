# 聚合接口数据源一致性

## 规则

聚合接口、图谱接口、列表接口、统计接口或映射补全逻辑中，响应实体集合和映射数据必须来自同一上游关系范围，除非代码、查询、接口契约或真实调用链证据证明两个来源的数据范围完全一致。

如果最终响应中的某个归属字段、状态字段或映射字段为空，优先怀疑实体集合来源和映射来源不一致。不要先添加兜底映射、替代字段或并行查询来掩盖源头不一致。

## 适用范围

- 后端聚合 API、图谱 API、列表 API、统计 API。
- 需要把实体集合与章节、归属、状态、进度、权限等映射数据拼装在一起的逻辑。
- 跨 Facade、跨服务或跨仓库取数后再组装响应的逻辑。

## 为什么

不同 Facade、Mapper、Repository 或 RPC 接口可能使用不同的数据范围、过滤条件和业务口径。即使参数名称相同，也不能推断它们返回的是同一批实体。

聚合层如果用来源 A 决定最终实体集合，却用来源 B 构建映射关系，就可能出现字段为空、错配或响应范围被悄悄扩大。正确修复通常是让最终实体集合、图谱参数、状态参数和响应映射来自同一个已验证的上游关系来源。

## 正例

- 先确认最终响应实体集合来自哪个上游关系。
- 再用同一实体集合或同一关系来源派生图谱参数、状态参数和映射表。
- 结论中写清失败点、最终赋值点、真实数据源、关键参数，以及为什么该修复足够。

## 反例

`queryEntityGraph` 中曾出现过数据源拆分风险：`graph2` 的实体集合来自 `sourceAFacade.listEntityByOwnerIds(ownerIds)`，而 `entityId -> ownerId` 映射来自 `sourceBFacade.getEntityOwnerMap(ownerIds)`。

这两个 Facade 可能返回不同实体范围，导致最终响应里的 `ownerId` 可能为空。此时不应通过补兜底查询来隐藏问题，而应先确认最终实体集合应该以哪个关系来源为准，并让映射数据与该实体集合闭环。

## 证据来源

- 失败点：最终响应字段赋值处，例如 `ownerId`、归属字段或状态字段被写入响应对象的位置。
- 实际数据源：决定最终响应实体集合的 Facade、Mapper、Repository 或 RPC 返回值。
- 关键参数：传入最终取数或计算点的参数，例如 `ownerIds`、`entityIds`、用户或组织范围参数。
- 分析要求：先找最终赋值点，再追踪真实下游调用链到最终 fetch 或 calculation 点，确认哪些参数实际影响结果，最后再决定最小修复。
- 修复边界：优先移除错误的数据源拆分；只有代码证据证明现有依赖确实缺失时，才考虑新增映射或兜底分支。
